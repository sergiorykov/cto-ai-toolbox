# How-to: план внедрения loop engineering в fullstack-монорепо

> [en: loop-engineering-case-monorepo-adoption.md](../loop-engineering-case-monorepo-adoption.md)

Практический план перехода на петли для конкретного сетапа:

> **Контекст.** Один монорепозиторий: Node.js фронт + бэк + PostgreSQL. Всё поднимается в
> `docker-compose`, разливается на контуры (dev/stage/…). Нужно, чтобы агент мог сам поднять
> контейнеры, поднять headless-браузер (Playwright) для фронта, прогнать тесты и итерировать до
> зелёного.

> **Оговорки о достоверности.** (1) Синтаксис команд Claude Code (`/goal`, `/loop`, `/schedule`)
> в 2026 меняется быстро — перед тем как оставлять что-то без присмотра, сверься с
> `code.claude.com/docs`. (2) Ниже — сборка практик из полевых гайдов (Developers Digest, Gaffer,
> Osmani), не из одного канонического источника. Проверяй версии пакетов и флаги.

---

## 0. Как звучит цель перехода (одним предложением)

> Перестать быть тем, кто вводит следующий промпт. Собрать в репозитории **harness** (обвязку),
> в котором агент по спецификации фичи сам пишет код, поднимает окружение в Docker, гоняет
> unit + e2e (Playwright headless), чинит по фидбеку от **независимого верификатора** и
> останавливается на зелёном билде или по лимиту итераций/бюджета — с PR на выходе.

Ключевая мысль всего направления: **петлю написать легко, трудная часть — верификатор внутри
неё.** Без независимой проверки петля не экономит труд, а быстрее производит неверный результат.

## 1. Три «глагола»: goal / loop / routine

Самая частая ошибка — свалить в одно слово три разных режима. Разведи их сразу:

| Глагол | Что значит | Claude Code | Где в вашем процессе |
|--------|-----------|-------------|----------------------|
| **Goal** | работать, пока цель не достигнута, потом стоп | `/goal <условие>` | «доведи фичу до зелёных тестов и чистого линта» |
| **Loop** | повторять, пока я рядом и смотрю | `/loop [интервал] <задача>` | «крути build-test-fix, пока я разбираю следующую задачу» |
| **Routine** | работать, пока меня нет | `/schedule <cron/webhook>` | «ночью прогоняй открытые PR, чини упавший CI» |

`/goal` важен не тем, что «крутится», а тем, **кто решает, что готово**: после каждого хода
Claude Code отправляет условие + транскрипт отдельной маленькой модели-судье (по умолчанию
Haiku), та возвращает yes/no с причиной. Это и есть «worker не ставит оценку сам себе».

Правило: начни с `/loop` (руками, на глазах) → как поверил верификатору, переводи в `/goal`
(само останавливается по условию) → как поверил бюджету, durable-вещи выноси в `/schedule`.

## 2. Что настроить в репозитории (фундамент harness)

Loop engineering по Osmani = 6 кирпичей: **automations, worktrees, skills, connectors,
sub-agents, memory**. Ниже — как каждый ложится в ваш монорепо.

### 2.1 `CLAUDE.md` (корень репо) — «конституция» агента
Что положить:
- как поднять стек: `docker compose -f docker-compose.yml -f docker-compose.test.yml up -d`;
- как накатить миграции и сиды; как дождаться готовности Postgres (healthcheck);
- команды проверок: `npm run lint`, `npm run typecheck`, `npm test`, `npm run test:e2e`;
- границы: что трогать нельзя (миграции прод-контура, секреты, `main`), что можно;
- порт-карта сервисов и правило изоляции портов для параллельных worktree.

### 2.2 Skills (`.claude/skills/…`) — процедурное знание один раз
Отдельные skill-файлы под повторяемые процедуры, чтобы агент не «угадывал»:
- `spin-up-stack` — поднять compose-профиль `test`, дождаться healthcheck, накатить миграции;
- `run-e2e` — установить браузеры (`npx playwright install --with-deps chromium`), поднять
  фронт, прогнать `npx playwright test`, собрать отчёт (junit/html);
- `db-reset` — пересоздать эфемерную тестовую БД под текущий worktree;
- `open-pr` — коммит в `claude/`-ветку, PR с описанием и ссылкой на отчёт тестов.

### 2.3 Sub-agents — паттерн maker/verifier (сердце петли)
Два разных агента (желательно **разные модели** — тогда «двое весов» должны согласиться):
- **maker** — пишет код по спецификации;
- **verifier** — свежим контекстом проверяет диф против spec + тестов, ловит то, что maker сам
  себе «простил». Не даёт закрыть задачу, пока не сойдётся.

Anti-pattern, который это убивает: **self-approving loop** — когда одна модель и делает, и
оценивает; она «удалит падающий тест и назовёт задачу выполненной».

### 2.4 Connectors (MCP) — руки и глаза агента
Минимальный набор под ваш стек:
- **Playwright MCP** — живой браузер как инструменты (`browser_navigate`, `browser_click`,
  `browser_snapshot` — читает accessibility-дерево, а не пиксели):
  `claude mcp add playwright npx @playwright/mcp@latest` (лучше пинить версию для
  воспроизводимости). Подробности по Docker/CI — раздел 4.
- **GitHub MCP/CLI** — Issues, PR, статусы CI как часть цепочки.
- **Postgres MCP** (опц.) — чтобы верификатор мог проверять состояние БД после сценария.

> Различай: **Playwright CLI** (`npx playwright test`) — твой регрессионный набор в CI;
> **Playwright MCP** — агент драйвит браузер вживую (self-QA, воспроизведение бага, генерация
> спеков). Нужны оба.

### 2.5 Worktrees — параллельность без коллизий
Каждому агенту — своя git-ветка и свой каталог (`git worktree`), чтобы параллельные фиксы не
затирали друг друга. Критично для вашего кейса: **у каждого worktree свой compose-проект и свои
порты/БД** (см. 4.3), иначе агенты подерутся за 5432/веб-порт.

### 2.6 Hooks — автоматический верификатор в точке коммита
Git-хук `post-commit`, который запускает ревью/тесты и кормит найденное обратно в фикс-петлю,
пока контекст «тёплый». Плюс stop-hooks, чтобы петля не завершалась раньше зелёного.

### 2.7 Memory — состояние вне одного диалога
`markdown`-файл (или доска в Linear/Asana): что сделано, что дальше, какие тесты были флаки.
Без этого каждый запуск CI выглядит «новым», агент переисследует уже затриаженный флак.

### 2.8 Budget/caps — до того, как уйти
Каждой `/goal` — оговорка «или стоп после N ходов»; каждой `/schedule` — дневной потолок трат.
«Петля без потолка радостно потратит все токены.»

## 3. Лестница верификации для вашего стека (от дешёвого к дорогому)

Верификатор должен быть **дешевле и надёжнее** проверяемого действия. Порядок применения:

1. **Детерминированное, бесплатное** — `tsc`/typecheck, `eslint`, компиляция, `docker build`.
2. **Unit-тесты** — `npm test` (Jest/Vitest), быстрый сигнал на бэке и утилитах фронта.
3. **Integration** — API против поднятой в compose Postgres (реальная БД, не мок).
4. **E2E** — Playwright headless против поднятого фронта: проверяет пользовательские сценарии.
5. **Model-as-judge / human gate** — только для того, что нельзя проверить детерминированно
   (UX-формулировки, архитектурные решения); дороже и хрупче, ставить последним.

Правило: чем ниже по лестнице удалось поймать ошибку, тем дешевле петля. E2E — не первый рубеж,
а последний автоматический.

## 4. Docker + Playwright + Postgres: конкретика

### 4.1 Отдельный compose-профиль для тестов
Заведи `docker-compose.test.yml`, поднимающий: `db` (Postgres с `healthcheck`), `backend`,
`frontend`, и job-контейнер `e2e` на официальном образе Playwright
(`mcr.microsoft.com/playwright` — браузеры и системные deps уже внутри, не ловишь
`error while loading shared libraries`).

Пример скелета (адаптируй под свои сервисы):

```yaml
# docker-compose.test.yml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: test
      POSTGRES_DB: app_test
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d app_test"]
      interval: 3s
      retries: 20
  backend:
    build: ./backend
    depends_on:
      db: { condition: service_healthy }
    environment:
      DATABASE_URL: postgres://postgres:test@db:5432/app_test
  frontend:
    build: ./frontend
    depends_on: [backend]
  e2e:
    image: mcr.microsoft.com/playwright:v1.latest  # пинить точную версию!
    depends_on: [frontend]
    working_dir: /work
    volumes: [".:/work"]
    command: ["npx", "playwright", "test", "--reporter=line,junit,html"]
```

### 4.2 Браузеры и headless
- В контейнере headless — дефолт, **не** форсь `--headed` (иначе «висит», ждёт дисплей).
- Если гоняешь Playwright не на официальном образе, ставь браузеры с системными deps:
  `npx playwright install --with-deps chromium` (без `--with-deps` — cryptic ошибки библиотек).
- Диагностика частых падений в CI:

| Симптом | Причина | Фикс |
|---|---|---|
| `Executable doesn't exist` | не установлены бинарники браузера | `npx playwright install --with-deps` |
| `error while loading shared libraries` | нет системных deps | `--with-deps` или образ `mcr.microsoft.com/playwright` |
| Server fails to start | Node < 18 | шаг с Node 18+ перед MCP |
| Висит вечно | ждёт дисплей | подтверди headless, не форси `--headed` |

### 4.3 Изоляция БД и портов на worktree (иначе параллельные петли дерутся)
- Уникальный `-p <project>` на каждый worktree: `docker compose -p wt_${BRANCH} …` — свои сети и
  контейнеры.
- Порты наружу не хардкодить, а маппить динамически (или гонять e2e внутри compose-сети, обращаясь
  к `frontend:PORT`, тогда наружу вообще не надо).
- Своя тестовая БД на прогон (`app_test_${BRANCH}`) + `db-reset` skill между прогонами.
- `storage state` для авторизованных сценариев: один раз
  `npx playwright codegen --save-storage=auth.json …`, дальше агент стартует уже залогиненным.

### 4.4 Память тестов (чтобы петля не переоткрывала флаки)
По умолчанию всё, что нашёл прогон, умирает с job. Храни отчёты между прогонами (junit/html как
CI-артефакты, или инструмент вроде Gaffer поверх). Тогда верификатор перед «починкой» сможет
спросить: этот тест падал на `main` неделю назад? он флаки? — и не чинить то, что чинить не надо.

## 5. Сквозной пример: петля проработки фичи

Возьмём фичу «добавить фильтр по статусу в списке заказов» (фронт + API + миграция).

**Шаг 0. Превратить запрос в строгую цель (goal-meta-skill).** Самый высокорычажный приём —
заставить агента сначала переписать задачу в точную цель: конечное состояние, как проверит, что
не трогать, когда стоп.

```
/goal перепиши мой запрос в точную спецификацию: конечное состояние, критерии приёмки,
как ты это проверишь (какие тесты), что нельзя трогать (миграции stage/prod, main),
и стоп-условие. Подтверди спецификацию, потом выполняй.
```

**Шаг 1. Build-test-fix как `/loop` (на глазах).**

```
/loop подними тестовый стек (docker compose -f docker-compose.test.yml up -d, дождись
healthcheck БД, накати миграции). Реализуй следующий пункт плана: миграция + API-эндпоинт +
UI-фильтр. После каждого изменения гоняй typecheck, lint, unit. Каждую ошибку возвращай
следующей инструкцией и чини. Стоп, когда билд зелёный и проверяющему нечего добавить.
```

**Шаг 2. E2E через Playwright (последний автоматический рубеж).**

```
подними frontend в compose, через Playwright MCP пройди сценарий: открой список заказов,
примени фильтр «в обработке», проверь, что показаны только заказы в этом статусе и счётчик
совпал. Если сценарий рабочий — сгенерируй spec в tests/e2e/orders-filter.spec.ts и
закоммить его.
```

**Шаг 3. Независимый verifier-subagent.**

```
/goal передай диф отдельному verifier-агенту (другая модель): проверить против спецификации
и тестов (unit + integration + e2e). Только PASS обеих сторон считается готовым. До 5 раундов.
Что verifier отверг дважды — вынеси мне. Стоп на первом чистом прогоне или на лимите.
```

**Шаг 4. Streak, а не один зелёный (для флаки-чувствительных мест).**

```
/goal прогони e2e против реалистичных сценариев. Любое падение сбрасывает счётчик. Готово
только после 3 подряд чистых прогонов. Один зелёный — это удача, серия — надёжность.
```

**Шаг 5. PR.** `open-pr` skill: коммит в `claude/orders-filter`, PR с описанием и приложенным
junit/html-отчётом. Ночная `/schedule`-рутина может дальше сама чинить упавший CI на этом PR.

## 6. План внедрения по фазам

**Фаза 1 (2–3 дня) — фундамент, без автономии.**
- Написать `CLAUDE.md` (команды стека, границы, порт-карта).
- Завести `docker-compose.test.yml` + healthcheck БД + сиды.
- Подключить Playwright MCP, прогнать руками self-QA один сценарий.
- Критерий готовности: агент по команде поднимает стек и гоняет unit+e2e локально.

**Фаза 2 (неделя 1) — первая петля на глазах.**
- Skills: `spin-up-stack`, `run-e2e`, `db-reset`, `open-pr`.
- Запустить build-test-fix как `/loop` на одной реальной некритичной фиче.
- Добавить verifier-subagent (другая модель). Проверить, что он реально что-то отклоняет.
- Критерий: одна фича доведена петлёй до зелёного и PR с минимумом твоих правок.

**Фаза 3 (недели 2–3) — изоляция и параллель.**
- Worktrees + изоляция compose-проекта/портов/БД на ветку.
- Хранение отчётов тестов (память), stop/`post-commit` hooks.
- Перевести доверенные петли из `/loop` в `/goal` с турн-капом.
- Критерий: 2–3 агента параллельно на разных фичах без коллизий по БД/портам.

**Фаза 4 (неделя 4+) — рутины и бюджет.**
- `/schedule`: ночной прогон открытых PR, автопочинка упавшего CI, ребейз протухшего.
- Дневные потолки трат; FinOps-обзор счёта за токены.
- Adversarial review (одна модель ревьюит PR другой) на merge-гейте.
- Критерий: утром — готовые/зелёные PR, спорное оставлено тебе; счёт под потолком.

## 7. Что важно настроить и где (сводка)

| Компонент | Где | Зачем |
|---|---|---|
| `CLAUDE.md` | корень репо | команды стека, границы, порт-карта |
| Skills | `.claude/skills/` | `spin-up-stack`, `run-e2e`, `db-reset`, `open-pr` |
| Sub-agents | конфиг агентов | maker/verifier на разных моделях |
| Playwright MCP | `claude mcp add playwright …` | живой браузер для self-QA и генерации спеков |
| `docker-compose.test.yml` | корень репо | изолированный тестовый стек + healthcheck БД |
| Playwright CI job | образ `mcr.microsoft.com/playwright` | headless e2e без возни с system deps |
| Worktrees | `git worktree` + `compose -p` | параллельные агенты без коллизий |
| Hooks | `.git/hooks` / `.claude/` | верификатор в точке коммита, stop-hooks |
| Memory + отчёты тестов | md-файл / junit-html артефакты | не переоткрывать флаки, помнить прогресс |
| Budget caps | в условии `/goal`, лимит `/schedule` | защита от runaway-счёта |

## 8. Анти-паттерны (держать в голове)

- **Self-approving loop** — maker сам себе судья. Лечится независимым verifier (другая модель).
- **Открытая петля без верификатора** — компаундинг ошибок, «галлюцинация прогресса», doom-loop
  (агент повторяет один и тот же сломанный подход с косметикой). Верификатор закрывает петлю.
- **Runaway cost** — нет капа → жжёт токены на задаче, что уже дважды провалил. Всегда турн-кап
  и дневной потолок.
- **Общая БД/порты на параллельных петлях** — гонки и «мигающие» падения. Изолируй по worktree.
- **`--headed` в контейнере** — зависание в ожидании дисплея. Только headless в CI.
- **Загрузка отчётов только при успехе (`if: success`)** — выбрасываешь ровно те данные (падения),
  что нужны следующему прогону. Грузить `if: always()`.
